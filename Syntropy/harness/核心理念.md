# Agent Harness —— 智能体的缰绳与连接器

> **一句话定义**：Agent Harness 是让单个智能体从「失控的野马」蜕变为「可编队、可问责、可沉淀」的交付体的系统性约束框架——它既不扼杀智能，也不放任蛮力，而是在创造力与确定性之间架设精确的桥梁。

---

## 一、从 Demo 到 Production：一个被忽视的鸿沟

今天的 AI 演示令人惊叹：一个 Agent 能写诗、能查资料、能调用工具。但当它从黑客松的展台走进真实的业务流程时，问题接踵而至：

- 两个 Agent 互相召唤，陷入无限递归，账单爆炸；
- Sub-Agent 输出冗长、发散，Orchestrator 无法聚合；
- 高风险工具被静默执行，直到事故酿成；
- 上下文窗口溢出，API 报 400，对话断裂；
- 三个月前的关键决策，Agent 早已「失忆」。

**根本症结在于：我们没有给 Agent 系上 Harness。**

没有 Harness 的 Agent 是聊天机器人——它取悦用户，却不交付确定性。
系上 Harness 的 Agent 是交付体——它在边界内自主，在失控前收敛，在沉默中留下可审计的足迹。

Syntropy（太和）正是为解决这一鸿沟而生。我们构建了**三层 Harness 架构**，用古代朝廷的隐喻做可视化外壳，用工程硬核做骨骼——让多智能体协作从「魔法」变成「机制」。

## 二、三层 Harness 架构

### 🎯 第一层：缰绳层（Reins Layer）
> *控制 Agent 间的连接与信息流动*

**核心命题**：当丞相传唤六部官员时，信息如何不失真、不冗余、不泛滥？

#### 2.1.1 结构化 ACP 协议：零损耗调度

传统 Multi-Agent 框架将结构化意图强行塞入自然语言 prompt，下游 LLM 再费力解析——**信息经历两次损耗**。

太和系统的 `Kernel.dispatch()` 直接传递结构化消息体：

```js
// server/core/Kernel.js
const result = await Promise.race([
    targetAgent.executeAsSubAgent({
        from: message.from,               // 谁发起的召唤
        instruction: message.payload.instruction,  // 精确指令
        depth: depth + 1,                 // 派生深度透传
        traceId: activeTraceId            // 链路追踪标识
    }),
    timeout
]);
```

`executeAsSubAgent` 在执行前临时注入 `[任务来源：{from}]` 上下文（`server/core/Agent.js:438`），执行后自动还原 system prompt。没有自然语言转换，没有信息二次损耗——**这是 Agent 协作的「二进制协议」**。

#### 2.1.2 并行批量调度：Promise.allSettled 的工业级应用

`call_officials` 技能（`skills/call_officials.js`）将 N 个串行任务压缩到单次最长任务的耗时：

```js
const settled = await Promise.allSettled(
    tasks.map(async (task, i) => {
        const response = await kernel.dispatch(message, depth, traceId);
        // ...
    })
);
```

- **理论加速比**：N 倍（并行而非串行）；
- **容错设计**：`allSettled` 确保单个官员失败不会阻塞整体流程；
- **感知透明**：`plan:preview` 事件让前端在执行前即展示完整作战图。

#### 2.1.3 官员响应契约：收敛 Sub-Agent 的「话痨」本能

LLM 在 sub-agent 模式下倾向于「展开对话」——对 Orchestrator 而言这是灾难。太和系统通过 system prompt 注入**强制收敛格式**：

```
**结论**：（一句话核心答案）
**要点**：（2-3条关键信息）
**备注**：（如有不确定，注明）
```

这不是 prompt engineering 的花招，而是**多智能体系统的接口契约**——就像 HTTP 的 JSON schema，它让异构 Agent 之间的互操作成为可能。

#### 2.1.4 派生深度控制：斩断无限递归

```js
// server/core/Kernel.js:109
const MAX_SPAWN_DEPTH = 2;
if (depth >= MAX_SPAWN_DEPTH) {
    return '[系统] 派生深度超限，拒绝执行';
}
```

`depth` 参数随每次 dispatch 透传累加。这不是防御式编程，而是**Harness 的底线规则**：Agent 可以协作，但协作必须有终止边界。

### 🛡️ 第二层：鞍座层（Saddle Layer）
> *保障 Agent 在边界内安全执行*

**核心命题**：当 Agent 拥有调用工具的能力时，谁来为它的行为负责？

#### 2.2.1 风险分级与御批审批

`SkillManager`（`server/runtime/SkillManager.js:62`）为每个技能标注 `riskLevel`（low / medium / high）。当 Agent 执行工具时，`handleToolCalls`（`server/core/Agent.js:314-334`）自动拦截：

```js
if (skill && (skill.riskLevel === 'medium' || skill.riskLevel === 'high')) {
    this.setStatus(AgentState.WAITING_FOR_HUMAN, `Waiting for approval: ${functionName}`);
    tracer.approvalWait(this.id, activeTraceId, functionName);
    this.kernel.events.publish('approval:request', {
        agentId: this.id,
        toolCallId: toolCall.id,
        functionName,
        args,
        riskLevel: skill.riskLevel
    });
    // 暂停执行，保存 pendingApproval
    return;
}
```

这不是简单的「人在回路」（Human-in-the-Loop）。太和系统实现了**基于风险等级的动态审批路由**：
- 低风险工具 → 自动执行，不打扰用户；
- 中高风险工具 → Agent 进入 `WAITING_FOR_HUMAN` 状态，前端弹出御批弹窗；
- 审批结果（含用户反馈）→ 作为上下文注入，Agent 无缝恢复执行。

状态机完整记录了这次中断（`server/core/Agent.js:375-428`），审批不是打断，而是**任务连续性的有机组成部分**。

#### 2.2.2 Dispatch 超时保护

```js
// server/core/Kernel.js:133-135
const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`[${message.to}] dispatch timeout after ${TIMEOUT_MS}ms`)), TIMEOUT_MS)
);
// TIMEOUT_MS = 60000
```

`Promise.race` 确保单个 sub-agent 的异常不会将 orchestrator 拖入永久挂起。60 秒是 Harness 的耐心上限——超过即熔断，返回异常，链路追踪记录失败标记。

#### 2.2.3 上下文安全压缩

`ContextManager.pruneContext()`（`server/runtime/ContextManager.js:96-168`）在 token 超限时不做粗暴截断，而是识别**安全删除单元**：

- 单个 user / assistant 消息 → 可独立删除；
- `assistant(tool_calls)` + 其所有对应 `tool result` → **视为原子组，只整组删除**。

这从根本上消除了「tool_call 丢失但 tool result 残留」导致的 400 API 错误——一个曾被无数 LLM 应用忽视的隐蔽陷阱。

#### 2.2.4 Tracer：可观测性即 Harness

`Tracer`（`server/infra/Tracer.js`）不是日志库，而是**Harness 的神经系统**：

- **traceId 传播**：每次用户指令生成根 traceId，子 Agent 继承并派生（`parent.depth` 格式）；
- **8 种诊断事件**：`agent.turn.start/end`、`tool.call.start/end`、`model.usage`、`dispatch.start/end`、`approval.wait`、`agent.stuck`；
- **日志脱敏**：正则自动截断 `sk-` 风格 API Key、Bearer Token、password 字段；
- **3 分钟卡死检测**：`setInterval` 巡检 `_activeAgents`，超阈值触发 `agent.stuck` 告警。

没有可观测性的 Harness 是盲人的缰绳——Tracer 让每一次失控都有迹可循。

### 🏛️ 第三层：马厩层（Stable Layer）
> *让 Agent 有记忆可归、有经验可沉淀*

**核心命题**：当 Agent 完成一次任务后，它的经验去了哪里？

#### 2.3.1 RRF 混合记忆检索

`MemoryManager`（`server/runtime/MemoryManager.js`）不满足于单一检索策略：

```js
async search(query, options = {}) {
    // 1. FTS5 全文检索（关键词精确匹配）
    const ftsResults = this.searchFTS(query, limit * 2);
    
    // 2. BGE 向量检索（语义相似度）
    const vectorResults = this.searchVector(queryVector, limit * 2);
    
    // 3. RRF 融合：score = Σ 1/(k + rank_i)
    return this.rankRRF(ftsResults, vectorResults, limit, 60);
}
```

BM25 负责「用户说税收我就找税收」的精确召回，向量检索负责「用户说财政收入我就联想到税收」的语义召回。**Reciprocal Rank Fusion（k=60）将两条独立溪流汇合成一条精准的记忆之河**——Recall@10 显著优于任一单一策略，且零额外 token 消耗。

底层 SQLite + FTS5 虚拟表 + 触发器自动同步 + Float32Array 向量存储（`server/runtime/MemoryManager.js:31-70`）——我们用 0 外部依赖实现了生产级混合检索。

#### 2.3.2 Session SQLite 持久化

`SessionStore`（`server/infra/SessionStore.js`）将会话历史从脆弱的 JSONL 迁移到 better-sqlite3：

```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    agent_id TEXT NOT NULL,
    role TEXT NOT NULL,
    content TEXT,
    tool_calls TEXT,
    tool_call_id TEXT,
    created_at INTEGER NOT NULL
);
CREATE INDEX idx_agent_created ON messages(agent_id, created_at);
```

- **性能不劣化**：索引查询 O(log n)，不随会话增长而衰减；
- **天然隔离**：按 `agent_id` 分区，一键清空单 Agent 历史；
- **原子性**：同步 API，无异步竞态。

#### 2.3.3 LLM 主动记忆

传统系统用关键词匹配触发记忆捕获——覆盖不全、误判率高、规则维护成本爆炸。

太和系统将决策权交给 Agent 本身。`save_memory` 技能（`skills/save_memory.js`）允许 Agent 在对话中**自主判断**哪些信息值得长期保存：

```js
handler: async ({ content, category, importance }, { agent }) => {
    await agent.memory.save(id, content, 'user_preference', {
        category,      // personal / preference / project / decision
        importance,    // high / medium / low
        savedBy: agent.id,
        savedAt: Date.now()
    });
}
```

Agent 不是被动接受记忆，而是**主动创造记忆**。零规则维护成本，利用 LLM 语义理解实现更准确的分类与去重（`MemoryManager.save()` 自动检测近 5 条同 role 重复内容）。

## 三、对比：Unharnessed vs Harnessed

| 维度 | Unharnessed Agent（聊天机器人模式） | Harnessed Agent（交付体模式） |
|------|-----------------------------------|------------------------------|
| **协作方式** | 自然语言接力，信息损耗不可控 | ACP 结构化协议，字段级精确传递 |
| **并发模型** | 串行阻塞，一个卡死全链断裂 | `Promise.allSettled`，容错并行 |
| **输出质量** | Sub-Agent 话痨，Orchestrator 崩溃 | 响应契约强制收敛，结论先行 |
| **递归控制** | 无边界，账单爆炸风险 | `MAX_SPAWN_DEPTH=2`，硬熔断 |
| **工具安全** | 静默执行，事后追责 | 风险分级 + 御批审批，事前拦截 |
| **超时保护** | 无限等待，进程僵死 | `Promise.race` + 60s 熔断 |
| **上下文管理** | 粗暴截断，API 频繁报错 | 安全单元原子删除，配对保护 |
| **记忆系统** | 无记忆或关键词规则匹配 | RRF 混合检索 + LLM 主动捕获 |
| **可观测性** | 分散日志，黑盒调试 | traceId 全链路 + 8 种诊断事件 + 卡死检测 |
| **持久化** | 内存易失，重启即失忆 | SQLite 会话持久化，经验可沉淀 |

> **关键洞察**：Unharnessed Agent 的核心问题是「它不知道自己不知道」——没有边界感知，没有失败收敛，没有经验积累。Harness 不改变 LLM 的能力上限，但它**将能力收敛到可预测、可问责、可复用的通道中**。

## 四、为什么 Harness 是 Agent 从 Demo 到 Production 的关键跃迁

### 4.1 Harness 不是限制，而是放大

给 Agent 系上 Harness，不是把它关进笼子，而是**给它一张可以全力奔跑的地图**。没有边界控制的自主是混乱，有边界约束的自主才是自由。`MAX_SPAWN_DEPTH=2` 不是让 Agent 变笨，而是让它敢于大胆协作——因为它知道系统会在深渊前拉住它。

### 4.2 可观测性是信任的基石

生产系统不接受「它有时候好用」。Tracer 的 traceId 全链路追踪、8 种结构化诊断事件、3 分钟卡死检测——这些不是监控奢侈品，而是**用户敢把业务流程交给 Agent 的前提条件**。没有 Harness 的 Agent 是不可审计的黑盒；有 Harness 的 Agent 是每一次决策都可回溯的透明体。

### 4.3 记忆是智能的复利

一个每次对话都从零开始的 Agent，无论 LLM 多强大，本质上都是「金鱼智能」。RRF 混合检索让经验可检索，Session 持久化让对话可延续，LLM 主动记忆让知识自生长——**这是从「单次交互」到「持续关系」的质变**。马厩层的真正意义不在于存储，而在于让 Agent 的每一次奔跑都让它变得更快、更稳、更懂你。

### 4.4 人机协同是生产的必要条件，不是可选功能

御批机制不是「为了安全而牺牲效率」的妥协。恰恰相反，它是**让 Agent 敢做高风险高价值任务的许可证**。没有审批拦截，Agent 只能被限制在低风险沙盒里打杂；有了 Harness 的鞍座层保护，Agent 才敢触碰真正改变业务的核心操作——因为系统证明了：最坏的情况，人类始终在场。

## 六、结语：Harness 是 Agent 时代的操作系统抽象

Syntropy（太和）用古代朝廷做隐喻，用三层 Harness 做骨架，用 Node.js + SQLite + Socket.io 做血肉。我们不是在做一个更酷的聊天机器人，而是在回答一个更根本的问题：

> **当多个智能体开始协作时，什么让它们值得被信任？**

答案是 Harness。缰绳层确保它们连接得有纪律，鞍座层确保它们执行得有边界，马厩层确保它们经验得有沉淀。

从 demo 到 production，从玩具到基础设施——
**Agent Harness，就是这道门槛的名字。**
