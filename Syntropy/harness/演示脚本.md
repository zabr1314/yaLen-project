# Syntropy (太和) 黑客松现场演示脚本

> **版本**: v1.0  
> **总时长**: ~4 分钟  
> **演示前提**: 前端 `npm run dev` (localhost:5173) + 后端 `node server/index.js` (localhost:3001) 已启动，浏览器全屏展示 Dashboard。

---

## 开场（0:00 - 0:30）

**🎤 演示者台词：**

> "各位评委好，我是 Syntropy（太和）项目的演示者。我们今天带来的是一个**可视化多智能体操作系统**。
>
> 大家可以看到，屏幕中央是一座 2D 像素沙盘——我们把它叫做**太和殿**。这里有 8 位 AI 官员：丞相统领调度，户部管数据，工部管工程，兵部管安全，还有史官、礼部、吏部、刑部各司其职。
>
> 用户以**皇帝**的身份在下方输入框下达指令，所有 Agent 的协作过程——思考、对话、审批、执行——都在沙盘上**实时可视化**。
>
> 接下来，我们用 **'修建江南水库'** 这个场景，演示一个复杂任务是如何被拆解、调度、协作、受控的。"

**🖥️ 屏幕画面：**
- 浏览器全屏展示 Dashboard，中央为 Phaser 3 沙盘（太和殿）
- 8 个像素小人分布在各自工位，状态灯为绿色 `idle`
- 左侧「百官录」显示 8 位官员的实时状态
- 右侧上方「奏折阁」为空，下方「记忆库」展示已有记忆条目
- 底部「诏令流水线」输入框等待输入

**⏱️ 预计耗时：** 30 秒

## Step 1：皇帝下达指令（0:30 - 0:50）

**🎤 演示者台词：**

> "第一步，皇帝下达指令。我们在底部输入框输入：'朕要在江南修建一座水库，需兼顾防洪、灌溉与生态，即刻拟旨。'
>
> 注意看——沙盘上的**皇帝气泡**会升起这行字，圣旨被自动路由给丞相处理。"

**🖱️ 操作动作：**
1. 鼠标点击底部金色输入框（获得焦点）
2. 键入：`朕要在江南修建一座水库，需兼顾防洪、灌溉与生态，即刻拟旨。`
3. 按 `Enter` 或点击金色 `Send` 按钮发送

**🖥️ 屏幕画面：**
- 输入框上方浮现「皇帝」气泡，显示刚输入的指令文字
- 气泡 3 秒后消散
- 左侧「百官录」中丞相状态从 `idle` 变为 `working`（蓝色呼吸灯）
- 右侧「奏折阁」新增一条「奉天承运第 XXXX 号」，状态为 **拟旨中**（紫色）
- 沙盘中央丞相小人头顶出现气泡："臣领旨..."

**⏱️ 预计耗时：** 20 秒

## Step 2：丞相拆解任务，展示 plan:preview（0:50 - 1:15）

**🎤 演示者台词：**

> "第二步，丞相收到圣旨后，开始**拆解任务**。大家看丞相的气泡——它在思考如何分解这个复杂工程。
>
> 展开右侧奏折阁的第一条记录，可以看到丞相生成的**执行计划**：先做地质与水文评估，再做工程预算，然后设计施工方案，最后部署安全监控。这就是我们的 `plan:preview`——复杂任务在动工前，先给你一个**全景蓝图**。
>
> 与此同时，丞相会根据计划，向户部、工部、兵部并行发出子任务。"

**🖱️ 操作动作：**
1. 点击右侧「奏折阁」第一条记录，展开详情
2. 点击「查看执行细节」，展示计划列表

**🖥️ 屏幕画面：**
- 丞相小人从议政厅走向大殿中央，气泡显示："拟将工程拆解为四步..."
- 展开后的奏折显示「执行计划」列表：
  - 户部：水文数据评估与预算编制
  - 工部：工程方案设计与知识检索
  - 兵部：安全风险评估与监控部署
- 左侧「百官录」中，户部、工部、兵部状态依次变为 `working`
- 诏令流水线状态从「拟旨中」变为「执行中」（蓝色）

**⏱️ 预计耗时：** 25 秒

## Step 3：并行调度户部 / 工部 / 兵部（1:15 - 1:45）

**🎤 演示者台词：**

> "第三步，**并行调度**。现在三个部门同时开工——户部在左中工位伏案计算预算，工部在右下工位查阅图纸，兵部在右上工位评估安全风险。

> 注意沙盘上的**寻路动画**：每位官员都会从自己的工位走向任务执行区域，执行时状态变为 `working`，头顶还会冒出当前的操作描述。这不是预录的动画，而是 Agent 真实状态的前端映射。
>
> 这就是 Syntropy 的核心：你不再面对一堆黑盒 API 调用，而是**看见**一个团队在协作。"

**🖥️ 屏幕画面：**
- 户部小人从左侧工位移向大殿中央，气泡："核算水文数据..."
- 工部小人从右下工位移向大殿中央，气泡："检索工程规范..."
- 兵部小人从右上工位移向大殿中央，气泡："评估安全风险..."
- 左侧「百官录」三个部门状态均为 `working`，终端输出显示各自的实时日志

## Step 4：工部检索知识库，展示 Memory Debugger（1:45 - 2:10）

**🎤 演示者台词：**

> "第四步，工部遇到一个问题——江南地区土质松软，需要参考历史工程案例。这时，工部会调用**记忆检索**。
>
> 我们打开右侧的「记忆库」，可以看到系统已经保存了大量历史决策和项目记录。点开工部的记忆透视镜，输入查询 '软土地基 水库 历史案例'——
>
> 大家看结果：系统通过 **FTS + Vector 混合检索**，找到了三条相关记忆，RRF 得分最高的是'太湖堤坝加固工程纪要'。工部基于这条记忆，修正了原设计方案，加入了桩基加固措施。"

**🖱️ 操作动作：**
1. 点击右侧「记忆库」顶部的「记忆透视镜 (RAG Debugger)」标签
2. 选择 Agent 为「工部」
3. 输入查询：`软土地基 水库 历史案例`
4. 按 `Enter` 执行搜索

**🖥️ 屏幕画面：**
- 工部小人头顶气泡："检索记忆：软土地基..."
- Memory Debugger 面板显示搜索结果：
  - Score: 0.8721 | FTS: 1 | VEC: 2
  - 内容摘要："太湖堤坝加固工程纪要：软土地基采用预应力混凝土桩..."
- 工部小人返回工位，气泡更新："方案已修正，增加桩基设计。"

**⏱️ 预计耗时：** 25 秒

## Step 5：兵部触发高风险工具，展示御批弹窗（2:10 - 2:35）

**🎤 演示者台词：**

> "第五步，也是最关键的一步——**风险控制**。

>
> 大家看屏幕中央弹出的这个**御批弹窗**：它清楚地告诉用户——谁（兵部）要做什么（`security_scan`）、风险等级是什么（HIGH）、参数详情是什么。用户作为'皇帝'，拥有最终裁决权：可以**准奏**，也可以**驳回**。"

**🖥️ 屏幕画面：**
- 兵部小人头顶气泡："请求执行安全扫描..."
- 屏幕中央弹出模态框「⚠️ 御批请求 (Imperial Approval)」：

  - 风险等级：**HIGH**（红色高亮）
  - 参数详情 JSON 展开
- 底部两个按钮：「驳回 (Reject)」红色 | 「准奏 (Approve)」金色
- 整个沙盘背景变暗，聚焦弹窗

**⏱️ 预计耗时：** 25 秒

## Step 6：用户驳回，兵部修正方案（2:35 - 3:00）

> 注意看：弹窗关闭后，兵部小人头顶气泡立刻变成'方案被驳回，重新评估...'。兵部不会死循环重试，而是会**理解反馈、修正参数**——比如缩小扫描范围，只针对内网段，再次提交。
>
> 这就是 Syntropy 的'御批'设计：不是简单的 Yes/No，而是**带反馈的迭代循环**。高风险操作必须经人确认，Agent 会根据人的意图自我修正。"

**🖱️ 操作动作：**
1. 鼠标点击弹窗左侧「驳回 (Reject)」按钮

- 若再次触发高风险操作，弹窗会再次弹出（本次演示中可假设修正后风险降为 MEDIUM，不再弹窗）

## 应急预案

> 现场演示不怕慢，就怕卡。如果某一步出现问题，按以下方式优雅过渡：

### 预案 A：后端响应延迟 / Agent 无反应

**现象**：输入指令后，丞相状态不变，无气泡，无日志。  
**应对**：
1. 保持微笑，说："看来丞相今日公务繁忙，我们手动唤醒一下。"
2. 打开左侧「百官录」，点击丞相卡片，检查连接状态。
3. 若后端确实卡住，直接说："为了节约时间，我们快进到这个状态——"然后展示一个**预先准备好的截图或录屏**（建议提前录制 30 秒关键流程备用）。
4. 过渡语："在实际运行中，这个过程只需几秒，我们来看下一步..."

### 预案 B：御批弹窗未触发

**现象**：兵部执行了工具，但没有弹出御批请求。  
**应对**：
1. 说："这次兵部评估认为风险等级在阈值以下，直接通过了。我们手动演示一下御批机制。"
2. 点击底部控制台的「内务府」→ 找到「模拟御批」或直接在输入框发送一条强制触发弹窗的测试指令（如 `@兵部 执行高危扫描 8.8.8.8/0`）。
3. 若仍不触发，提前准备的截图/录屏兜底。

### 预案 C：Memory Debugger 搜索无结果

**现象**：工部查询记忆，返回"未找到相关记忆"。  
**应对**：
1. 说："这说明当前知识库还没有相关记录——正好展示 Syntropy 的**知识沉淀**能力。"
2. 切换到「记忆库」标签，展示已有记忆条目（项目、偏好、决策分类）。
3. 说："随着系统运行，每次对话和决策都会自动存入记忆库，越用越聪明。我们来换个已有记录的查询词..."
4. 换一个更通用的查询词（如 `预算`、`项目`），确保有结果返回。

### 预案 D：Phaser 沙盘黑屏 / 渲染异常

**现象**：中央沙盘全黑或角色不显示。  
**应对**：
1. 这是视觉问题，不影响核心功能演示。
2. 说："沙盘暂时休息，我们来看控制台的核心能力。"
3. 将演示重心迅速切换到「百官录」「奏折阁」「记忆库」三个面板，强调**结构化的 Agent 管理、完整的决策链路、可检索的记忆系统**——这三点才是评分重点。

### 预案 E：总时间不够（剩余 < 30 秒）

**应对**：
1. 立即跳到 **Step 7（丞相汇总）** 的截图/录屏。
2. 说："由于时间关系，我们直接看结果——所有子任务已并行完成，决策链清晰可追溯。"
3. 直接念**收尾台词**，确保评委听到核心价值主张。
