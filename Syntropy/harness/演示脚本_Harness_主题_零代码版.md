# Syntropy 3 分钟 Harness 主题演示脚本（零代码版）

> **总时长**: 180 秒，零容忍超时  
> **核心主题**: Harness —— 不是模型有多强，而是你怎么把模型、工具、上下文和反馈机制真正"拴"在一起  
> **演示策略**: 单场景贯穿 Harness 三层（缰绳层 → 马厩层 → 鞍座层），真实运行 + 决策链路收尾  
> **生死线**: 所有操作基于现有系统真实能力，绝不赌未实现的功能

---

## 第一幕：开场定调 —— 什么是 Harness（0:00 - 0:25）【25秒】

**🎤 台词：**
> "各位评委好，这是 Syntropy。
>
> 今天我不展示一个更强的模型，我要展示的是 **Harness**。
>
> Harness 原本是缰绳。在 AI 语境里，它指的不是单个模型有多强——而是你怎么把模型、工具、上下文和反馈机制**真正拴在一起**，让 Agent 从能聊天，变成能稳定干活、持续交付的系统。
>
> 3 分钟，看 Harness 的三层骨架：缰绳层怎么调度，马厩层怎么记忆，鞍座层怎么约束。"

**🖥️ 屏幕画面：**
- 全屏沙盘，8 个像素小人 idle 绿色
- 左侧百官录全绿

- **不要操作**，让画面自己说话

**⏱️ 时间控制：** 说最后一句话时，手放到键盘上准备输入

## 第二幕：皇帝下旨 ——  Harness 第一层：缰绳层（0:25 - 0:50）【25秒】

**🖱️ 操作动作：**
1. 点击底部输入框
2. 键入：`评估江南水库工程的预算、技术与合规`
3. 按 Enter 发送

**🎤 台词（输入时同步说）：**
> "皇帝下旨。注意看丞相——"

**🖥️ 屏幕画面：**
- 输入框上方升起「皇帝气泡」
- 丞相状态 idle → thinking（蓝色呼吸灯）
- 左侧百官录丞相状态变化
- 右侧「奏折阁」新增记录，状态「拟旨中」

**🎤 台词（丞相 thinking 时）：**
> "——这是 Harness 的**缰绳层**。丞相不是在转发 prompt，而是在执行结构化 ACP 协议：拆解意图、收敛格式、控制派生深度。信息不经自然语言转译，零损耗调度。"

**⏱️ 时间控制：**
- 如果丞相 10 秒内没有出 plan preview，立刻点击奏折阁展开预热记录：

## 第三幕：并行调度 ——  Harness 缰绳层的核心（0:50 - 1:35）【45秒】

**🖥️ 屏幕画面（理想情况）：**
- 丞相 plan preview 出现：户部查预算、工部查技术、刑部查合规
- 左侧百官录：户部、工部、刑部同时变为 working（黄色/橙色）
- 沙盘上三个小人同时走向大殿中央，头顶气泡显示各自任务
- 奏折阁状态变为「执行中」

**🎤 台词（核心输出时间，必须背熟）：**
> "传统 Multi-Agent 是串行的——A 说完 B 说，一个卡死全链断。
>
> Syntropy 用 **Promise.allSettled** 实现真正的并行调度。户部算预算、工部看图纸、刑部审合规——**同时开工**，单个失败不阻塞全局。
>
> 更重要的是约束：每位官员的响应被强制收敛为**结论 + 要点 + 备注**，就像 API 的 JSON Schema。这叫**响应契约**——没有它，Sub-Agent 的话痨会让 Orchestrator 崩溃。
>
> 还有**派生深度控制**：`MAX_SPAWN_DEPTH = 2`。Agent 可以协作，但协作必须有终止边界。Harness 不是限制创造力，是给创造力一张可以全力奔跑的地图。"

**🎯 关键技巧：**
- 这 45 秒 LLM 在跑，你**必须说话填满**
- 手指放在「奏折阁」上，plan preview 一出现立刻点击展开
- 如果某个 Agent 卡住超过 20 秒，指着他说："这位官员延迟了，但 allSettled 保证其他人继续"——体现容错设计

**⏱️ 时间控制：**
- 如果 1:20 三个部门还没全部完成，直接进入下一幕：
  > "时间关系，我们来看 Harness 的下一层——"

## 第四幕：记忆检索 ——  Harness 第三层：马厩层（1:35 - 2:05）【30秒】

**🖥️ 屏幕画面：**
- 工部小人气泡显示检索相关内容
- 或工部已返回 idle，丞相正在汇总

**🎤 台词：**
> "三部门结果即将聚合。但注意工部——它在查资料时，自动检索了历史案例。"

**🖱️ 操作动作：**
1. 点击「内务府」→ 选择「工部」→ 打开「记忆透视镜」
2. 输入：`软土地基 水库`
3. 按 Enter

**🖥️ 屏幕画面：**
- 搜索结果出现："太湖堤坝加固工程纪要..."
- 显示 RRF 得分、FTS 排名、Vector 排名

**🎤 台词：**
> "这是 Harness 的**马厩层**。RRF 混合检索——BM25 关键词 + BGE 向量语义，Reciprocal Rank Fusion 将两条独立溪流汇成精准的记忆之河。零外部依赖，一个 SQLite 搞定。
>
> 没有马厩层的 Agent 是金鱼智能——每次对话从零开始。Harness 让经验可检索、可沉淀、可复用。"

**⏱️ 时间控制：**
- 如果搜索无结果（预热没做好）：
  > "这说明当前知识库需要沉淀——但系统正在自动捕获记忆。"
  - 切到「记忆库」展示已有条目，继续说："每次对话和决策都会自动存入，越用越聪明。"

## 第五幕：决策链路 + 收尾升华 ——  Harness 第二层：鞍座层（2:05 - 3:00）【55秒】

**🖥️ 屏幕画面：**
- 丞相汇总完成，奏折阁状态变「已办结」
- 左侧百官录全绿 idle

**🎤 台词：**
> "任务完成。但 Harness 还没有结束——"

**🖱️ 操作动作：**
1. 点击奏折阁中刚完成的记录旁的「决策链」按钮
2. 展示完整的决策树：皇帝 → 丞相 → 并行调度 → 户部 / 工部 / 刑部 → 工部记忆检索 → 汇总回复

**🖥️ 屏幕画面：**
- 决策链路树状展开，每个节点清晰可见
- traceId 贯穿全链路

**🎤 台词（语气加重）：**
> "这是 Harness 的**鞍座层**——可观测性即 Harness。
>
> 不是分散的日志，不是黑盒调试。每一个决策、每一次调度、每一个工具调用，都被 traceId 贯穿，形成**可回溯的决策链路**。
>
> 如果某个环节触及高风险操作——比如兵部请求安全扫描——系统不会静默执行。Agent 会进入 WAITING_FOR_HUMAN 状态，弹出御批弹窗，参数透明，人类始终握有最终缰绳。
>
> 3 分钟，我们展示了 Harness 的三层骨架：
> - **缰绳层**——结构化调度、并行执行、响应契约、递归控制
> - **马厩层**——RRF 混合记忆、经验沉淀、持续学习
> - **鞍座层**——全链路可观测、风险分级、人机协同
>
> 市面上大多数 Agent 框架是黑盒。Syntropy 把这一切——
> **可视化、结构化、可审计。**
>
> 一句话：Syntropy 让 AI 协作，像治理朝廷一样透明、有序、可控。
>
> 赋予 AI 以秩序，赋予 Multi-Agent 以 Harness。
> 谢谢各位评委！"

**🖥️ 屏幕画面：**
- 最后一句话时，保持决策链路树展开，或切回沙盘全景
- **不要操作鼠标**，让画面定住，鞠躬

## 生死应急预案（按优先级排序）

### 预案 0：LLM 响应慢（最可能）
**现象**：丞相 thinking 超过 15 秒，或并行调度超过 60 秒没完  
**应对**：
- 不要沉默！立刻开始讲 Harness 理念填满时间
- 超过 30 秒还没动静："由于现场网络，我们来看刚才跑完的结果——"，点击预热的奏折记录展示 plan preview 和决策链
- 过渡语："在实际运行中，这个过程只需几秒，Harness 的并行调度确保效率——"

### 预案 1：丞相没有并行调度（只派了一个部门）
**现象**：只有工部或户部一个部门 working  
**应对**：
- 调整台词："丞相判断该任务需要深度调研，先派工部检索历史案例—— Harness 的调度不是无脑并行，而是根据任务依赖关系动态选择并行或串行。"
- 等工部完成后，自然过渡到第四幕记忆检索
- **不强行提并行，把重点转向 Harness 的智能调度策略**

### 预案 2：Memory Debugger 搜索无结果
**现象**：工部查询记忆，返回"未找到相关记忆"  
**应对**：
- "这说明当前知识库还没有相关记录——正好展示 Harness 的**知识沉淀**能力。"
- 切换到「记忆库」标签，展示已有记忆条目（项目、偏好、决策分类）
- "随着系统运行，每次对话和决策都会自动存入记忆库，越用越聪明。"

### 预案 3：决策链按钮无法点击 / 无内容
**现象**：奏折阁记录没有决策链，或点击后空白  
**应对**：
- "决策链路需要任务完全结束后生成——由于时间关系，我给大家展示 Harness 鞍座层的另一个维度。"
- 切到「百官录」，点击任意 Agent 卡片，展示终端输出和状态历史
- "这就是可观测性：每一个 Agent 的每一次思考、每一次工具调用，都被完整记录。"

### 预案 4：总时间不够（剩余 < 40 秒）
**应对**：
- 立即跳到第五幕的收尾台词，从"3 分钟，我们展示了 Harness 的三层骨架"开始念
- 保证金句出口："赋予 AI 以秩序，赋予 Multi-Agent 以 Harness"

### 预案 5：系统完全崩溃
**应对**：
- 保持微笑："看来我们的 Agent 今天也想休息。"
- 打开手机播放提前准备好的完整录屏
- 对着录屏讲解 Harness 三层架构，语速放慢

## 评委最可能问的 3 个问题 & 回答

### Q1："Harness 和普通的 Multi-Agent 框架有什么区别？"
**标准回答：**
> "普通框架解决的是'怎么让多个 Agent 说话'，Harness 解决的是'怎么让它们值得被信任'。区别在三层：缰绳层确保连接有纪律，马厩层确保经验有沉淀，鞍座层确保执行有边界。没有 Harness 的 Agent 是聊天机器人，有 Harness 的 Agent 是交付体。"

### Q2："御批弹窗这次没有触发，是不是说明风险控制是摆设？"
**标准回答：**
> "恰恰相反。本次评估任务没有触及高风险阈值，审批未被触发——但这正是 Harness 的精髓：约束不是每次都要表现出来，而是**始终在场**。就像安全带，最好的状态是你永远不需要感受它绷紧的那一刻。当风险真的出现时，系统会 100% 拦截。"

### Q3："生产环境怎么部署？"
**标准回答：**
> "单 Node.js 进程 + SQLite，零 Docker、零 Redis、零向量数据库。我们现在用的是 DeepSeek，成本是 GPT-4o 的 1/20。SQLite 可以无缝迁移到 PostgreSQL，向量索引可以切到 Milvus——接口已经隔离好了。"
