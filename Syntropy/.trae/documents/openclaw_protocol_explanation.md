# OpenClaw 协议与端口机制解释及优化计划

## 1. 核心解释：为什么是 `ws://` 但端口也是 18789？

您观察到的 `http://127.0.0.1:18789/` 和配置中的 `ws://localhost:18789` 其实指向的是**同一个服务**。

### 技术原理
OpenClaw Gateway（以及大多数现代实时 Web 服务）采用的是 **协议升级 (Protocol Upgrade)** 机制：
1.  **同一个端口 (18789)**：同时监听 HTTP 和 TCP 流量。
2.  **HTTP 请求**: 当您在浏览器访问 `http://127.0.0.1:18789` 时，服务器返回 HTML/JS 页面（控制台 UI）。
3.  **WebSocket 升级**: 当我们的 Adapter 连接时，它首先发送一个特殊的 HTTP 请求（携带 `Upgrade: websocket` 头）。服务器收到后，会将连接从 HTTP 协议“升级”为 WebSocket 协议 (`ws://`)。
4.  **持久连接**: 升级完成后，这条连接就变成了一条双向的、实时的 WebSocket 通道，用于接收 Agent 的实时思考和状态流。

**结论**: 您的理解是对的，端口确实是同一个。配置写 `ws://` 是为了明确告诉 Adapter 我们预期建立的是 WebSocket 长连接，而不是请求一次就断开的 HTTP 短连接。

## 2. 优化计划：自动协议转换

为了消除这种困惑并提升用户体验，我建议修改代码，允许用户直接输入 `http://` 开头的地址，由系统自动处理协议转换。

### 2.1 修改目标文件
- `server/lib/openclaw/adapter.js`

### 2.2 修改逻辑
在 `OpenClawGatewayAdapter` 的配置加载或连接逻辑中增加智能转换：
- 如果用户输入 `http://...` -> 自动转换为 `ws://...`
- 如果用户输入 `https://...` -> 自动转换为 `wss://...`
- 如果用户输入 `ws://` 或 `wss://` -> 保持不变

### 2.3 预期效果
您在前端设置界面直接粘贴浏览器地址栏的 `http://127.0.0.1:18789`，点击保存，系统自动识别并成功连接，无需手动修改协议头。

## 3. 实施步骤
1.  **修改 Adapter 代码**: 更新 `resolveOriginForUpstream` 或在构造函数中处理 URL 规范化。
2.  **验证**: 使用 `http://` 地址测试连接是否成功。
