# OpenClaw 体验优化计划：奏折阁重构与丞相动画升级

## 1. 目标
优化“天命系统”与 OpenClaw 的交互体验，重点解决以下问题：
1.  **信息断层**: 奏折阁 (`MemorialsPanel`) 和 起居注 (`LogSidebar`) 未显示 OpenClaw 的回复内容。
2.  **动画干扰**: 丞相移动时，气泡显示“移动中...”覆盖了重要的对话内容。
3.  **视觉优化**: 重新设计奏折阁 UI，使其更符合“天命”主题，并清晰展示对话流。

## 2. 奏折阁与起居注重构 (`MemorialsPanel.tsx` & `LogSidebar.tsx`)

### 2.1 数据源问题
目前 `CourtStore` 中的 `Decree` 对象包含 `logs` 数组，但前端组件可能没有正确渲染 `logs` 中的回复内容，或者样式不明显。

### 2.2 重构方案
- [x] **奏折阁 (`MemorialsPanel`)**:
    - **改为对话流模式**: 不再只是显示一个静态的“内容”和“计划”，而是像聊天记录一样展示 [皇帝] 和 [丞相] 的对话。
    - **高亮回复**: OpenClaw 的回复应有明显的视觉区分（如使用不同的背景色或边框）。
    - **状态可视化**: 清晰展示当前奏折是“拟旨”、“执行中”还是“已办结”。
- [x] **起居注 (`LogSidebar`)**:
    - **实时流**: 保持现有的流式日志，但优化排版，确保长文本回复可读性。
    - **过滤**: 移除无意义的系统级日志（如“移动到...”、“状态更新...”），专注于对话和关键决策。

## 3. 丞相动画优化 (`Agent.ts`)

### 3.1 问题分析
`Agent` 类在移动时（`moveTo`）或状态更新时，可能会调用 `say('移动中...')`。如果此时丞相正在说话（`say('微臣以为...')`），移动指令会打断并覆盖掉重要的话。

### 3.2 优化策略
- [x] **优先级控制**: 给 `say` 方法增加 `priority` 参数。
    - `low`: 状态描述（“移动中”、“思考中”）。
    - `high`: 对话内容（OpenClaw 回复）。
    - **规则**: `low` 优先级的消息不能覆盖 `high` 优先级的消息。
- [x] **持久化对话**: 对话内容应该保持显示，直到被下一句对话覆盖，或者超时（超时时间应根据文本长度动态计算）。
- [x] **移动静默**: 除非状态是 `idle` -> `moving` 且当前无对话，否则移动时不强制显示气泡。

## 4. 实施步骤

### 4.1 修改 `Agent.ts`
- 修改 `say(message, duration, priority)` 方法。
- 在 `update` 或 `preUpdate` 中移除自动显示“移动中”的逻辑，或者降低其优先级。

### 4.2 修改 `MemorialsPanel.tsx`
- 重新设计 `Decree` 卡片的内部布局，增加 `ConversationView` 子组件。
- 渲染 `decree.logs` 时，筛选出 `actor` 为 `Minister` (丞相) 或 `Emperor` (皇帝) 的记录，按时间轴展示。

### 4.3 修改 `LogSidebar.tsx`
- 优化渲染逻辑，支持多行文本，增加头像或角色标识。

## 5. 立即执行
先从 `Agent.ts` 的气泡逻辑入手，解决“丞相乱说话”的问题，然后重构 UI。
