# 计划：Syntropy (太和) 深度开发路线图

## 目标
根据对现有代码库的分析，识别并规划出 **Syntropy (太和)** 项目的后续开发方向。目标是将系统从 MVP 阶段推进到更成熟、更具交互性和可扩展性的 **Beta 版本**。

重点关注：
1.  **Agent 认知升级**：优化记忆检索与压缩机制。
2.  **前端交互增强**：提升 RAG 可视化与沙盘互动性。
3.  **系统稳定性**：完善测试与自动化演示脚本。

## 现状分析
- **Agent 能力**：
    - 记忆系统目前基于简单的 SQLite FTS5 + 暴力向量搜索，缺乏长时记忆的压缩与整理机制 (`onSleep` 仅为占位符)。
    - 工具调用虽然有“天牢”拦截，但缺乏复杂的任务链编排能力。
- **前端交互**：
    - 沙盘 (Phaser) 仅用于展示，缺乏拖拽指派等深度交互。
    - RAG 检索过程是黑盒，用户无法直观看到 Agent 检索到了哪些文档片段。
- **开发者体验**：
    - 缺乏自动化测试与演示脚本，手动触发“天牢”拦截流程较为繁琐。

## 拟定开发计划 (按优先级排序)

### 1. 认知层升级：RAG 检索可视化与记忆整理 (High Priority)
**目标**：打破 Agent 思考的“黑盒”，让记忆检索过程透明化，并优化长时记忆效率。

-   **后端 (Backend)**:
    -   **实现 `onSleep` 记忆压缩**: 在 [Agent.js](file:///Users/hongyulin/Desktop/天命系统/server/core/Agent.js) 中实现 `onSleep` 钩子。当 Agent 进入休眠时，自动调用 LLM 对当日对话进行摘要，提取关键事实并存入 `summary` 表。
    -   **检索调试接口**: 新增 API `/api/agent/:id/memory/debug`，允许前端输入 Query，返回 RRF 排序后的原始文档片段及相似度得分。

-   **前端 (Frontend)**:
    -   **RAG 调试器组件**: 在“内务府”或 Agent 详情页增加一个“记忆透视镜”面板。
    -   **可视化展示**: 使用颜色编码（如热力图）展示关键词匹配 (FTS) 与语义匹配 (Vector) 的权重分布。

### 2. 交互层增强：沙盘互动与环境反馈 (Medium Priority)
**目标**：让 2D 沙盘不仅仅是装饰，而是真正的交互界面。

-   **拖拽指派 (Drag & Drop)**:
    -   在 [MainScene.ts](file:///Users/hongyulin/Desktop/天命系统/src/game/MainScene.ts) 中实现 Agent 的拖拽逻辑。
    -   当用户将“奏折”图标拖拽到某个 Agent 身上时，自动触发任务分发。

-   **动态环境反馈**:
    -   **忙碌状态特效**: 当系统负载高或 Agent 正在进行高强度计算（如复杂推理）时，沙盘中的“服务器”机柜出现过载火花特效。
    -   **昼夜交替**: 根据系统时间自动切换沙盘的色调（白天/黑夜模式）。

### 3. 工程化完善：自动化演示脚本 (Low Priority, but useful for Demo)
**目标**：一键展示系统的核心亮点（如天牢拦截），无需手动构造复杂场景。

-   **编写 Mock 脚本**:
    -   创建 `scripts/demo_jail.js`，模拟一个名为 "rogue_agent" 的 Agent。
    -   该脚本会自动发送一条违规指令（如 `delete_database`），触发后端的风险拦截。
    -   前端自动弹出“御批”窗口，演示完整的人机协同流程。

## 验证计划
-   **功能验证**:
    -   运行 `onSleep` 逻辑，检查数据库中是否生成了对话摘要。
    -   使用 RAG 调试器输入测试 Query，确认能看到检索结果及其得分。
    -   运行演示脚本，确认“天牢”拦截机制正常触发。

## 下一步行动
建议首先执行 **Task 1 (RAG 检索可视化与记忆整理)**，因为这直接提升了系统的核心竞争力——**可解释性 (Explainability)**。
