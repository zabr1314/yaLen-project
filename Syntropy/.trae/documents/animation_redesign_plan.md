# 天命系统动画重演计划 (Court Animation Redesign)

陛下，鉴于朝廷组织架构已调整为“一帝一相、六部一史”，微臣将重新编排朝堂之上的行为动画逻辑，使百官的行动更加符合新的身份与职责。

## 1. 核心设计理念
*   **皇帝 (Emperor)**：不动如山，或御驾亲征。皇帝的行动由**用户直接控制**，不再自动跑来跑去。
*   **丞相 (Prime Minister)**：百官之首，上传下达。丞相将频繁往返于“议政厅”与“太和殿”（向皇帝汇报）之间。
*   **六部尚书 (Six Ministries)**：各司其职，动静结合。
    *   接旨时：前往议政厅听令。
    *   执行时：前往各自工位（或中央区域）办事。
    *   闲暇时：在各自区域待命或闲聊。
*   **史官 (Historian)**：如影随形。每当有重大事件（执行/审计）发生，史官应前往现场记录，事毕回角落存档。

## 2. 详细剧本 (CourtService.ts)

### 场景一：颁布诏令 (Drafting -> Planning)
*   **旧制**：丞相直接开始规划。
*   **新制**：
    1.  **用户（皇帝）** 点击发布诏令。
    2.  **丞相** 从议政厅快步走到太和殿前（皇帝面前）。
    3.  **丞相** 播放 `working` 动画（模拟接旨/记录）。
    4.  **丞相** 回到议政厅，开始 `planning`。

### 场景二：规划与修正 (Planning)
*   **动作**：丞相在议政厅原地 `working`（思考）。
*   **修正**：如果皇帝驳回，丞相再次跑到太和殿前听取“御批”，然后再回议政厅修改。

### 场景三：刑部审核 (Reviewing)
*   **旧制**：御史走到丞相身边。
*   **新制**：
    1.  **刑部尚书** 从刑部（右侧）走到议政厅（丞相身边）。
    2.  二人面对面，播放 `working` 动画（激烈讨论）。
    3.  审核完毕，刑部尚书回刑部，丞相等待。

### 场景四：分发任务 (Approved -> Dispatching)
*   **新制**：
    1.  诏令通过。
    2.  **丞相** 走到大殿中央。
    3.  **对应部门尚书**（如兵部）从工位跑到大殿中央。
    4.  丞相交付任务（交互动画），尚书领命。

### 场景五：执行与记录 (Executing)
*   **新制**：
    1.  **尚书** 回到各自工位（或特定执行点，如兵部去机房，礼部去对外接口）。
    2.  **史官** 从角落跑到尚书身边，拿出小本本记录（`working` 动画）。
    3.  执行结束，尚书 `idle`，史官跑回角落存档。

### 场景六：最终审计 (Auditing)
*   **新制**：
    1.  **刑部尚书** 前往执行部门工位进行验收。
    2.  验收通过后，向丞相/皇帝复命。

## 3. 实施步骤

### 3.1 坐标点位精修 (`LOCATIONS`)
*   定义 `THRONE` (龙椅/皇帝位置)。
*   定义 `CENTER_STAGE` (大殿中央，分发任务用)。
*   优化六部工位坐标。

### 3.2 动画逻辑重写 (`CourtService.processDecree`)
*   按照上述剧本，分阶段重写 `processDecree` 中的 `setTargetPosition` 和 `setAgentStatus` 逻辑。
*   引入更细腻的 `delay` 控制，确保动画连贯（例如：等尚书跑到中央，丞相再开始说话）。

### 3.3 史官自动化 (`HistorianObserver`)
*   将史官的行动逻辑封装，使其能自动跟随“当前活跃的执行者”。
