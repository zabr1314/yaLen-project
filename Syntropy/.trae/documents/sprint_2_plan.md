# AI 帝国·数字中枢 MVP 演进计划 (Sprint 2: 动态交互与事件系统)

既然视觉和基础控制（Sprint 1 & 1.5）已经稳定，接下来的重点应该从“静态展示”转向“动态交互”。我们需要让这个数字中枢看起来像是一个**正在实时运行的系统**，而不仅仅是手动点击的玩偶。

## 核心目标
1.  **引入随机事件系统**：模拟真实的业务场景（如服务器负载波动、用户请求激增、数据库死锁），让 Agent 自主响应。
2.  **实现 Agent 间的协作**：展示 Agent 之间的交互（例如：Minister 派活给 Engineer，Engineer 完成后汇报）。
3.  **增强 UI 信息量**：Sidebar 不再只是日志，应该有实时的系统监控面板（仪表盘）。

## 详细实施步骤

### 1. 随机事件生成器 (Event Generator)
-   **目标**：模拟不可预测的外部环境。
-   **实现**：
    -   创建一个 `GameEventManager`（可能是新的 Store slice 或独立服务）。
    -   定时（或随机）生成事件，例如：
        -   `Server Overload` (服务器过载)：需要 Engineer 去机房（地图上的新区域）处理。
        -   `New Policy Request` (新政策请求)：需要 Minister 去办公桌起草文件。
    -   事件发生时，地图对应区域闪烁红光/黄光，并产生全局通知。

### 2. 智能体协作流 (Agent Workflow)
-   **目标**：展示 Multi-Agent 的价值。
-   **实现**：
    -   **场景：故障修复流程**
        1.  系统触发“服务器过载”警告。
        2.  Minister 接收到警报，移动到 Engineer 身边，气泡显示“去修服务器！”。
        3.  Engineer 收到指令，移动到机房区域，头顶进度条（Working...）。
        4.  Engineer 完成工作，状态恢复，Minister 记录日志“故障已修复”。
    -   **技术点**：需要实现简单的状态机（State Machine）来管理 Agent 的行为序列（Idle -> Moving -> Interacting -> Working -> Reporting）。

### 3. 系统监控仪表盘 (Dashboard UI)
-   **目标**：让 Sidebar 看起来更像一个运维/指挥中心。
-   **实现**：
    -   在 Sidebar 顶部增加可视化图表（可以使用简单的 CSS 进度条或 SVG）。
        -   **CPU 负载**：动态变化的百分比。
        -   **在线人数**：随机波动的数字。
        -   **待处理工单**：显示当前积压的任务数。
    -   当数值超过阈值（如 CPU > 90%）时，自动触发上述的“随机事件”。

### 4. 地图扩展 (Map Expansion)
-   **目标**：配合新事件增加功能区。
-   **实现**：
    -   **机房 (Server Room)**：在地图左下角绘制服务器机柜（闪烁的 LED 灯效果）。
    -   **会议室 (Meeting Room)**：在地图中间绘制一张大圆桌，用于 Agent 碰头。

## 预期效果
完成 Sprint 2 后，用户不再需要手动点击按钮来驱动 Agent。他们只需要看着屏幕，就能看到一场由 AI 主导的“自动化运维大戏”：系统报警 -> 丞相指挥 -> 工程师干活 -> 系统恢复。这将极大地增强“天命系统”的叙事感和智能化体现。
